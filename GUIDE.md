# GUIDE.md — как это устроено и как воспроизвести с нуля

Этот документ — подробное объяснение технической стороны проекта: что из чего
состоит, почему выбраны именно такие решения, и пошаговая инструкция, по которой
можно поднять всё с нуля на чистом сервере. В конце — список реальных проблем,
с которыми мы столкнулись, и как их решили (полезно, если наткнётесь на то же).

Если нужна просто короткая шпаргалка по командам — см. [README.md](README.md).
Здесь — с объяснениями "почему" и "что будет, если пойдёт не так".

## 1. Обзор: что за система и как данные текут по ней

Задача: собирать объявления о продаже машин из двух разных источников (сайт
drom.ru и телеграм-каналы), приводить их к одному виду, показывать аналитику
и — бонусом — предсказывать цену по ссылке на объявление.

```
                    ┌─────────────┐        ┌──────────────┐
  drom.ru  ───────► │ drom_ingest │───────►│  drom_raw    │ (сырой JSON,
                    │  (DAG,      │        │  (Postgres)  │  как есть)
                    │  каждые     │        └──────┬───────┘
                    │  10 минут)  │               │
                    └─────────────┘               │
                                                   ▼
                    ┌─────────────┐        ┌──────────────────────┐
  Telegram ───────► │telegram_    │───────►│ normalize_           │
  (через прокси,    │ingest (DAG) │        │ car_listings (DAG)   │
  если заблокирован)└─────────────┘        │ — приводит оба       │
                    ┌──────────────┐  ───► │   источника к одному │
                    │telegram_raw  │       │   виду (regex/LLM)   │
                    │ (Postgres)   │       └──────────┬───────────┘
                    └──────────────┘                  │
                                                       ▼
                                              ┌──────────────────┐
                                              │  car_listings    │  ◄── общая
                                              │  (Postgres)      │      таблица
                                              └────────┬─────────┘
                                       ┌────────────────┴────────────────┐
                                       ▼                                 ▼
                              ┌─────────────────┐             ┌──────────────────┐
                              │    Superset      │             │ train_price_model │
                              │  (дашборд по     │             │  (DAG, раз в     │
                              │   car_listings)  │             │  неделю) → модель │
                              └─────────────────┘             └─────────┬────────┘
                                                                          │
                                                                          ▼
                                                              ┌────────────────────┐
                        ссылка на объявление ───────────────►│  prediction-api     │
                                                              │  (FastAPI /predict) │
                                                              │  парсит ссылку ТЕМ  │
                                                              │  ЖЕ кодом, что и    │
                                                              │  normalize, гоняет  │
                                                              │  через модель       │
                                                              └────────────────────┘
```

Ключевая идея, которая держит всё вместе: **один и тот же код** (`common/parsers.py`)
превращает и сырые строки из БД, и одиночную ссылку пользователя в один и тот же
"единый вид" (`brand, model, year, price, mileage, region, source, url, collected_at`).
Поэтому сервис предсказания гарантированно видит те же признаки, на которых
обучалась модель.

## 2. Почему так устроено (архитектурные решения)

**Почему Airflow, а не просто cron-скрипты?**
Нужны были: расписание, повторные попытки при сбое, видимость (какой таск когда
упал и почему — прямо в UI), и возможность запустить руками для отладки. Cron
всё это даёт хуже или не даёт вообще.

**Почему один Postgres на всё (данные + метаданные Airflow + метаданные Superset)?**
Команда — два человека, сервер — один. Три БД внутри одного Postgres-инстанса
(`cars`, `airflow`, `superset`) — это одна установка, один бэкап, один пароль
администратора, а не три отдельных сервиса, за которыми надо следить.

**Почему drom парсится HTML-скрейпингом, а не через API?**
У drom.ru нет публичного API для объявлений. Карточка объявления и страница
объявления — это структурированный HTML с атрибутами `data-ftid="..."` — по ним
и цепляемся (см. раздел 4.1 про конкретные селекторы).

**Почему telegram сначала пробовали через Telethon, а в итоге — через LLM
поверх веб-превью?**
Telethon логинится как реальный пользователь Telegram — для headless-сервера
нужна заранее сгенерированная сессия (интерактивный логин), и это привязывает
пайплайн к чьему-то личному аккаунту. Публичное веб-превью `t.me/s/<канал>`
отдаёт последние посты БЕЗ авторизации вообще — этого достаточно, раз каналы
публичные. А для самого разбора текста (какая тут марка/цена/пробег) обычные
регулярки регулярно спотыкались о неформальный человеческий текст (см. раздел
6.5 — там реальный пример, как адрес в подписи объявления был принят за цену).
LLM с этим справляется на порядок лучше и стоит копейки (доли рубля за пост).

**Почему цена в модели — `log(цена)`, а не сама цена?**
Цены на машины в датасете разбросаны на 3 порядка — от 40 тысяч (Лада 90-х) до
29 миллионов (Mercedes G-Class 2025). Если учить модель на сырых рублях, она
будет минимизировать АБСОЛЮТНУЮ ошибку — и тогда одна дорогая машина, предсказанная
неточно на пару миллионов, "весит" в функции потерь как тысяча дешёвых Лад,
предсказанных неточно на 50 тысяч. Модель на логарифме учится предсказывать
ОТНОСИТЕЛЬНУЮ ошибку — это то, что реально нужно ("ошиблись на 20%", а не
"ошиблись на 2 миллиона").

**Почему `common/parsers.py` — один файл на DAG и на сервис?**
Прямое требование задания: "сервис парсит именно эту ссылку тем же кодом, что и
в шаге 2". Технически это работает через то, что `docker-compose.yml` монтирует
`./common` как volume и в контейнеры Airflow, и в контейнер prediction-api —
физически это один файл на диске, не две копии.

## 3. Пошаговое воспроизведение с нуля

### 3.1. Сервер

Любой Linux-сервер (мы использовали Ubuntu 22.04), минимум 2 CPU / 4GB RAM
(Airflow + Superset прожорливые). Публичный IP нужен, если хотите открытые
порты наружу (пункт 3.7 — но их можно вообще не открывать и ходить только
через SSH-туннель).

```bash
sudo apt update && sudo apt install -y docker.io docker-compose-plugin git
sudo systemctl enable --now docker
```

### 3.2. Код и `.env`

```bash
git clone <repo-url> car-price && cd car-price/project_drom_teleg
cp .env.example .env
```

Заполняем `.env` — вот что означает каждая переменная и откуда её взять:

| Переменная | Откуда взять |
|---|---|
| `POSTGRES_PASSWORD` | любой сгенерированный пароль (`openssl rand -hex 16`) |
| `AIRFLOW_UID` | `id -u` на сервере (на Linux; если 0/root — так и ставим) |
| `AIRFLOW_FERNET_KEY` | `python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"` |
| `AIRFLOW_SECRET_KEY`, `SUPERSET_SECRET_KEY` | `python3 -c "import secrets; print(secrets.token_hex(32))"` |
| `TELEGRAM_CHANNELS` | публичные каналы с объявлениями, без `@`, через запятую |
| `TELEGRAM_API_ID/HASH` | **можно оставить пустыми** — тогда сбор пойдёт через публичное веб-превью, без Telethon |
| `LLM_API_KEY/BASE_URL/MODEL` | ключ любого OpenAI-совместимого провайдера (мы использовали polza.ai + `deepseek/deepseek-v4-flash` — дёшево и достаточно умно для этой задачи) |
| `TELEGRAM_PROXY` | пусто, пока не убедитесь, что Telegram у вас не заблокирован (см. 3.4) |

### 3.3. Поднять стек

```bash
docker compose up -d --build
docker compose ps   # все сервисы должны быть Up/healthy
```

Если что-то не поднялось — почти всегда причина в `postgres` (остальные
сервисы от него зависят и просто ждут). Смотрите `docker compose logs postgres`.

### 3.4. Проверить, не заблокирован ли Telegram у хостера

Это реальная проблема, с которой мы столкнулись — многие хостинг-провайдеры
(особенно в РФ) режут трафик до серверов Telegram на уровне сети. Проверка:

```bash
curl --max-time 8 https://t.me/s/some_public_channel
# если зависает/таймаутит, а обычные сайты открываются — Telegram заблокирован
```

Более точная диагностика:
```bash
ping -c 3 149.154.167.99          # IP одного из серверов Telegram
traceroute 149.154.167.99         # если пакеты пропадают на каком-то хопе
                                   # без единой ошибки — это похоже на DPI-блокировку
```

Если заблокирован — нужен прокси. Мы подняли `xray-core` (клиент VLESS+Reality)
как ещё один сервис в docker-compose, который слушает SOCKS5 на порту 1080
**только для контейнеров нашей сети** (наружу не торчит):

```bash
cp xray/config.json.example xray/config.json
```

В `config.json` нужно вписать данные из своей VLESS-подписки (`address`, `port`,
`id` пользователя, `publicKey`, `serverName` — это всё есть в vless://-ссылке
или JSON-конфиге, который выдаёт ваш VPN-провайдер на протоколе VLESS+Reality):

```json
{
  "inbounds": [{"listen": "0.0.0.0", "port": 1080, "protocol": "socks",
                "settings": {"auth": "noauth", "udp": true}}],
  "outbounds": [{"protocol": "vless", "settings": {"vnext": [{
      "address": "ваш-vless-сервер.example.com", "port": 443,
      "users": [{"id": "UUID-пользователя", "encryption": "none",
                 "flow": "xtls-rprx-vision"}]
  }]}}, "streamSettings": {"network": "tcp", "security": "reality",
      "realitySettings": {"serverName": "example.com", "fingerprint": "firefox",
                           "publicKey": "ваш-публичный-ключ", "show": false}}}]
}
```

Поднимаем и проверяем, что прокси реально пробивает блокировку:
```bash
docker compose up -d xray
docker run --rm --network <project>_default curlimages/curl:latest \
  --socks5-hostname xray:1080 -o /dev/null -w '%{http_code}\n' https://t.me/s/some_channel
# должно быть 200
```

Если 200 — прописываем в `.env`:
```
TELEGRAM_PROXY=socks5h://xray:1080
```
и пересоздаём контейнеры, которые ходят в Telegram:
```bash
docker compose up -d airflow-scheduler airflow-webserver prediction-api
```

### 3.5. Первый прогон сбора данных

```bash
docker compose exec airflow-scheduler airflow dags unpause drom_ingest
docker compose exec airflow-scheduler airflow dags unpause telegram_ingest
docker compose exec airflow-scheduler airflow dags unpause normalize_car_listings

docker compose exec airflow-scheduler airflow dags trigger drom_ingest
docker compose exec airflow-scheduler airflow dags trigger telegram_ingest
# подождать ~10-30 секунд, пока таски отработают, потом:
docker compose exec airflow-scheduler airflow dags trigger normalize_car_listings
```

Проверить результат:
```bash
docker compose exec postgres psql -U <POSTGRES_USER> -d <POSTGRES_DB> \
  -c "select source, count(*) from car_listings group by source;"
```

Дальше DAG'и сами продолжат собирать данные каждые 10-15 минут — ничего
руками трогать больше не нужно.

### 3.6. Дашборд в Superset

Заходим в Superset (порт 8088 или через nginx на 80), логинимся (`admin` /
пароль из `.env`), дальше руками:

1. **Settings → Database Connections → + Database** — тип Postgres, строка
   подключения: `postgresql+psycopg2://<POSTGRES_USER>:<POSTGRES_PASSWORD>@postgres:5432/<POSTGRES_DB>`
   (хост `postgres` — это имя сервиса в docker-сети, не `localhost`!).
2. **Datasets → + Dataset** — выбрать БД, схему `public`, таблицу `car_listings`.
3. **Charts → + Chart** — на датасете `car_listings` собрать несколько штук:
   - Bar chart: group by `brand`, метрика `AVG(price)` — средняя цена по маркам
   - Bar chart: group by `region`, метрика `COUNT(id)` — объявлений по регионам
   - Bar chart: group by `source`, метрики `COUNT(id)` и `AVG(price)` отдельно —
     сравнение drom vs telegram
   - Table/Bar: group by `brand, model`, метрика `COUNT(id)` — топ моделей
4. **Dashboards → + Dashboard** — назвать, перетащить туда все чарты, Save.

(Всё это можно сделать и через REST API Superset вместо кликанья — см.
`common`-скрипты в истории коммитов репозитория, если нужна автоматизация;
но для разового сетапа руками быстрее.)

### 3.7. Обучить модель и проверить `/predict`

```bash
# минимум 30 строк в car_listings, иначе упадёт с понятной ошибкой
docker compose exec prediction-api python train.py
```

Скрипт выведет метрики на отложенной выборке (MAE/MedAE/RMSE/MAPE — что каждая
из них означает и почему они могут "скакать" — см. раздел 5). Модель сохраняется
в `model_store/price_model.joblib` (volume, шарится между `airflow-*` и
`prediction-api`).

Проверка сервиса:
```bash
curl -X POST http://<server>:81/predict \
  -H "Content-Type: application/json" \
  -d '{"url":"https://auto.drom.ru/<город>/<марка>/<модель>/<id>.html"}'
```
В ответе — `predicted_price`, распарсенные `features`, и если у объявления есть
цена — `real_price` + `verdict` ("переоценено/недооценено на N%").

### 3.8. Закрыть лишние порты

По умолчанию `docker-compose.yml` уже настроен так, что наружу торчат только
`nginx` (80 → Superset, 81 → prediction-api) — остальные порты (`5432`, `8080`,
`8088`, `8010`) привязаны к `127.0.0.1` (см. `ports: - "127.0.0.1:PORT:PORT"` в
файле). Если у вас более старая версия без этого — просто добавьте `127.0.0.1:`
перед каждым портом кроме `nginx`.

Для доступа к Airflow UI с рабочей машины — SSH-туннель:
```bash
ssh -L 8080:localhost:8080 <server>
# дальше http://localhost:8080 на своей машине
```

Обычный `ssh -L` иногда отваливается (сеть моргнула, сервер перезагрузился).
Чтобы туннель не приходилось поднимать руками каждый раз, полезно завернуть
его в `autossh` + скрипт-обёртку, который перезапускает даже сам `autossh`,
если тот вдруг решит завершиться (подробности и объяснение — почему просто
`autossh` иногда недостаточно — в разделе 6.9).

## 4. Как устроен каждый компонент

### 4.1. `dags/drom_ingest.py` — сбор с drom.ru

Тянет объявления по городам (список в `DROM_CITIES`) через `requests`+`BeautifulSoup`,
без браузера/Selenium — drom отдаёт обычный серверный HTML.

Ключевые детали:
- **Кодировка** — страницы в `windows-1251`, а не `utf-8`. `requests` сам не
  угадывает, поэтому кодировка задаётся явно: `resp.encoding = "windows-1251"`.
- **Пагинация — путевая, не query-параметром.** `auto.drom.ru/<город>/all/page2/`
  работает, а `?page=2` **молча игнорируется и отдаёт первую страницу** (см.
  раздел 6.3 — там как это обнаружили).
- **Селекторы карточки** — по атрибутам `data-ftid`, не по CSS-классам (классы
  у drom минифицированы и меняются от сборки к сборке, `data-ftid` стабильнее):
  `[data-ftid="bulls-list_bull"]` (карточка), `a[data-ftid="bull_title"]`
  (ссылка+заголовок), `[data-ftid="bull_price"]` (цена), `[data-ftid="bull_description-item"]`
  (список характеристик, включая пробег).
- **Анти-бот вежливость** — случайная пауза 2-5 секунд между запросами, ротация
  4 разных User-Agent, ретраи с экспоненциальным бэкоффом на 429/5xx. Это
  снижает риск, но не убирает целиком — при резком увеличении числа городов/страниц
  drom всё равно может начать отдавать 429 (см. раздел 6.4).
- **Дедуп** — `ON CONFLICT (url) DO UPDATE` в `drom_raw` (нужен `UNIQUE INDEX
  ON drom_raw(url)`), чтобы повторные прогоны не копили дубли одного и того же
  объявления, а обновляли снимок (цена/пробег могли измениться).

### 4.2. `dags/telegram_ingest.py` — сбор из Telegram

Два режима, выбираются автоматически функцией `_collect()`:
1. **Telethon** (user-API) — если заданы `TELEGRAM_API_ID/HASH` и есть
   `.session`-файл. Может читать полную историю и приватные каналы.
2. **Веб-превью `t.me/s/<канал>`** — если API-доступов нет. Работает только с
   публичными каналами и только с последними постами на странице превью
   (обычно последние ~20), зато вообще без авторизации.

Для страницы превью каждый пост — блок `<div class="tgme_widget_message"
data-post="channel/12345">`; текст — в `.tgme_widget_message_text`, дата — в
`.tgme_widget_message_date time[datetime]`.

### 4.3. `common/parsers.py` — общая нормализация

Два входа:
- `normalize_drom(raw)` — `raw` это то, что сложил `drom_ingest` (JSON карточки).
  Марка/модель/город достаются **из URL** (`.../<город>/<марка>/<модель>/<id>.html`)
  — это надёжнее, чем парсить заголовок, потому что URL латиницей и всегда
  одного формата. Цена и пробег — из структурированных полей карточки
  (`price_text`, `desc_items`), а не regex по всему тексту.
- `normalize_telegram(raw)` — сначала пробует LLM (`common/llm.py`), при неудаче
  откатывается на регулярки (`extract_price`, `extract_brand_model`, и т.д.).

Отдельная функция `parse_listing_url(url)` — то, что вызывает сервис
предсказания: по домену определяет источник (drom.ru или t.me) и вызывает
`_fetch_drom`/`_fetch_telegram_post` + соответствующий `normalize_*`. Для
одиночного telegram-поста НЕ нужен Telethon — тот же трюк с публичным
веб-превью, что и в батч-сборе.

### 4.4. `common/llm.py` — LLM-извлечение

Отправляет текст поста (обрезанный до 4000 символов) в OpenAI-совместимый
эндпоинт с системным промптом "извлеки brand/model/year/price/mileage/region
в JSON". Спроектирован так, чтобы быть полностью необязательным и безопасным:
- Пустой `LLM_API_KEY` → сразу возвращает `{}`, никаких запросов.
- Любая ошибка (сеть, таймаут, кривой JSON, модель обернула ответ в список
  вместо объекта) → ловится, логируется, возвращает `{}`. Вызывающий код
  (`normalize_telegram`) в этом случае откатывается на регулярки — пайплайн
  никогда не падает из-за LLM.

### 4.5. `dags/normalize_car_listings.py` — свод в одну таблицу

Читает **весь** `drom_raw` и `telegram_raw` при каждом запуске (не только новое
с прошлого раза — это ОК, потому что дальше идёт upsert по `url`, лишняя
повторная обработка просто ничего не меняет для уже виденных строк).

Каждая строка обрабатывается в своём `SAVEPOINT`
(`conn.begin_nested()`) — если конкретная строка не смогла нормализоваться
(битый JSON, неожиданный формат), откатывается только её сохранение, а не весь
прогон целиком.

### 4.6. `prediction_service/train.py` / `dags/train_price_model.py` — обучение

Одна и та же логика продублирована в двух местах намеренно: DAG должен уметь
работать автономно (не звать внешний скрипт), а `train.py` — для ручного
запуска без Airflow. **Важно:** если правите логику обучения — правьте оба
файла, синхронизация ручная.

Модель — `GradientBoostingRegressor` из sklearn, признаки: марка/модель/регион
(one-hot) + год/пробег (как есть). Обёрнута в `TransformedTargetRegressor`
(учится на `log1p(цена)`, `predict()` всё равно возвращает рубли — inverse-
transform происходит внутри автоматически).

### 4.7. `prediction_service/app.py` — сервис предсказания

FastAPI, три ручки: `GET /health`, `GET /` (HTML-форма), `POST /predict`.
Модель грузится лениво при первом запросе и кэшируется в памяти процесса —
**если переобучили модель, но процесс сервиса не перезапускали, он всё ещё
использует старую модель из памяти**, пока не перезапустится сам (падение,
деплой) или явно (`docker restart prediction-api`).

### 4.8. `nginx/nginx.conf` — точки входа

Два `server` блока: порт 80 проксирует на `superset:8088`, порт 81 — на
`prediction-api:8000`. Это ровно то, что просит задание — "один сервис на
сервере с двумя точками входа".

## 5. Как читать метрики модели

Скрипт обучения печатает:
```
метрики на отложенной выборке (N строк): MAE=... руб, MedAE=... руб, RMSE=... руб, MAPE=...%
```

- **MAE** (средняя абсолютная ошибка) и **RMSE** — в рублях, но при разбросе
  цен на 3 порядка (десятки тысяч — десятки миллионов) сильно зависят от того,
  попала ли в тестовую выборку пара действительно дорогих редких машин
  (Lamborghini, G-Class и т.п.) — модель на них почти всегда ошибается на
  большую абсолютную сумму просто потому, что таких машин мало и учиться на
  них особо не на чем. Это ожидаемо, а не признак сломанного пайплайна.
- **MedAE** (медианная ошибка) — устойчива к таким редким случаям, показывает
  типичную ошибку на "обычном" объявлении. Более информативна на этом датасете.
- **MAPE** — средняя ошибка в процентах. Была особенно нестабильна ДО перехода
  на `log1p`-обучение (могла улетать за 500-900%, если пара цен в датасете
  были близки к нулю или сильно занижены) — после перехода на логарифм упала
  до разумных ~40%.

Если метрики выглядят подозрительно — сначала проверяйте не "удалить выбросы",
а конкретные строки: часто то, что выглядит выбросом статистически, при
ручной проверке оказывается нормальным реальным объявлением (старая убитая
машина за копейки, редкий люксовый экземпляр задорого) — см. пример в разделе
6.6, как мы такое перепроверяли.

## 6. Реальные проблемы, с которыми столкнулись, и как их решили

Этот раздел — конспект того, что реально ломалось при разработке этого проекта.
Если у вас всплывёт то же самое — вот готовые ответы.

### 6.1. Superset REST API: `GET /api/v1/dataset/` возвращает пусто, хотя объект только что создан

**Симптом:** создаёшь database/dataset через API с JWT-токеном (`Authorization:
Bearer ...`), запрос на создание возвращает 200 с айдишником, но следующий
`GET` того же объекта — `404 Not found`, хотя запись реально есть в БД
(проверяется напрямую через psql).

**Причина:** в этой версии Superset JWT-аутентификация не до конца
прокидывает `current_user` в проверку прав доступа (`can_access_all_datasources`),
из-за чего запрос с Bearer-токеном обрабатывается как будто от анонимного
пользователя без прав — соответственно фильтр по правам обрезает все объекты.

**Решение:** логиниться не через `/api/v1/security/login` (JWT), а как обычный
браузер — `GET /login/` (достать `csrf_token` из HTML-формы), `POST /login/`
с логином/паролем, сохранить cookie сессии, дальше использовать её для всех
запросов к API. С сессионной cookie права проверяются правильно.

### 6.2. Модель, обученная в Airflow-контейнере, не грузится в prediction-api

**Симптом:** DAG `train_price_model` отработал успешно, сохранил
`.joblib`-файл, но `POST /predict` падает с `500` и в логах —
`ModuleNotFoundError: No module named 'dill'`.

**Причина:** контейнеры `airflow-*` и `prediction-api` собираются из РАЗНЫХ
Dockerfile с разным набором пакетов. Что-то в окружении Airflow заставило
`joblib`/`pickle` при сериализации использовать `dill` — а в образе
`prediction-api` пакета `dill` нет, распаковать такой файл он не может.

**Решение:** обучать модель тем же процессом/образом, который её потом
использует — `docker compose exec prediction-api python train.py`, а не через
DAG, если сразу нужно смотреть результат в сервисе. (Оба способа обучения
одинаково валидны для эксплуатации — но если сталкиваетесь с этой ошибкой,
просто переобучите через `prediction-api`.)

### 6.3. Пагинация drom: `?page=N` тихо не работает

**Симптом:** DAG собирает объявления с 3 "разных" страниц, но `car_listings`
почти не растёт от прогона к прогону.

**Диагностика:**
```bash
curl -s "https://auto.drom.ru/all/?page=2" -o p_qs.html
curl -s "https://auto.drom.ru/all/page2/"   -o p_path.html
curl -s "https://auto.drom.ru/all/"         -o p1.html
# сравнить множества ссылок на объявления в каждом файле
```
Оказалось: ссылки в `p_qs.html` совпадают на 100% со страницей 1 (drom просто
игнорирует `?page=`), а в `p_path.html` — совсем другой набор, ноль пересечений
со страницей 1.

**Решение:** пагинация — частью пути (`/all/page2/`, `/all/page3/`, ...), а не
query-параметром.

### 6.4. Массовый разовый досбор упирается в антибот (429)

**Симптом:** при попытке собрать сразу много данных (например, расширив список
городов с 8 до 30+ для одного разового прогона) после первых 5-6 городов
начинают идти сплошные `429 Too Many Requests`, ретраи с бэкоффом не помогают.

**Что сделали:** не стали городить более хитрый антибот-обход, а просто дали
скрипту дособирать то, что получится (иногда ретраи всё же пробивались) и
приняли, что для большого разового набора данных нужно либо больше времени
(лимит обычно отпускает через 10-30 минут), либо более редкие города за
несколько отдельных прогонов, а не всё разом.

**Важный урок по коду:** если пишете отдельный скрипт для разового массового
сбора (не через штатный DAG), **пишите в БД инкрементально** (после каждого
города/страницы), а не копите всё в памяти и не сохраняйте одним большим
запросом в самом конце — если процесс упадёт или его придётся прервать
посередине, при накопительном подходе вы потеряете вообще всё, а не только
необработанный хвост.

### 6.5. Регулярка цены путает адрес в тексте с ценой

**Симптом:** у двух совершенно разных постов в telegram (один — объявление о
машине, другой — вообще не объявление, а анонс) `extract_price` вернул
одинаковую подозрительно маленькую цену — 6000 рублей.

**Диагностика:** в обоих текстах был один и тот же адрес: *"Боровское шоссе,
д. 6, к. 3"*. Регулярка на "число + суффикс тысяч" (`(\d+[.,]?\d*)\s*(?:тыс|к\b)`)
допускала повисшую запятую в числе (`[.,]?` без обязательной цифры после), и
"6," перед "к." (от "корпус") матчилось как "6 тысяч".

**Решение:** ужесточили регулярку — десятичный разделитель принимается, только
если сразу за ним идёт цифра (`(?:[.,]\d+)?` вместо `[.,]?\d*`). Заодно
добавили проверку разумного диапазона (30 000 — 100 000 000 руб) КО ВСЕМ
веткам функции, а не только к последней — если совпадение оказалось за
пределами диапазона, теперь код пробует остальные способы найти цену вместо
того, чтобы сразу вернуть подозрительное значение.

### 6.6. "Выбросы" в датасете — почти всегда не ошибки

Когда после сбора 1700+ строк метрики модели выглядели "слабыми", первым
порывом было — "давай удалим выбросы". При тщательной проверке (по каждому
подозрительному значению смотрели исходные данные, а не просто статистику)
выяснилось:
- `model = '2107'`, `'911'`, `'308'` и т.д. — это НЕ мусорный текст, а реальные
  названия моделей (Lada 2107, Porsche 911, Peugeot 308) — в автомобильной
  индустрии числовые обозначения моделей — норма, не аномалия.
- Машина 1990-х годов за 50-60 тысяч рублей с пробегом 250+ тысяч км — это не
  ошибка парсинга, а реальная старая убитая машина.
- Подержанный люксовый внедорожник за несколько миллионов старше 30 лет — при
  проверке сырых данных с сайта оказалось, что это ровно та цена, которую
  указал продавец (подготовленный для бездорожья экземпляр или просто
  завышенный ценник — так тоже бывает на реальных площадках).

**Вывод:** прежде чем удалять "выброс" — проверьте исходный текст/страницу
объявления вручную. Статистическая аномалия в мультибрендовом датасете с
разбросом цен на 3 порядка — это норма, а не сигнал об ошибке. Настоящей
ошибкой из полутора тысяч строк оказалась только одна — пробег `999999` км
(это стандартное значение-заглушка/потолок одометра, а не реальный пробег).

### 6.7. `docker compose up -d <сервис>` пересоздаёт лишние контейнеры

**Симптом:** меняешь `.env` (например, добавляешь новую переменную) и
перезапускаешь только один сервис (`docker compose up -d prediction-api`), а
compose заодно пересоздаёт `postgres`, `superset` и другие, которые вроде бы
не должны были измениться.

**Причина:** compose считает хэш конфигурации каждого сервиса, и он включает
не только сам сервис, но иногда и результат `env_file: .env` целиком —
изменение файла `.env` в принципе может задеть хэш нескольких сервисов сразу,
даже если конкретной переменной, которая изменилась, у них в `environment:`
нет.

**Что делать:** не бояться (именованные volume'ы, в т.ч. `postgres_data`,
переживают пересоздание контейнера — данные не теряются), но **предупреждать
второго человека в команде** перед таким рестартом, если он в этот момент
активно работает в Airflow UI или Superset — соединение на секунды оборвётся.
После пересоздания `postgres` не забудьте перезапустить `nginx` (см. 6.8).

### 6.8. nginx отдаёт `502 Bad Gateway` после пересоздания backend-контейнеров

**Симптом:** `superset`/`prediction-api` пересоздались (новый IP в docker-сети),
`nginx` не трогали — а он вдруг начал отвечать `502` на оба входа.

**Причина:** `nginx` резолвит имя `superset`/`prediction-api` в IP при старте
и держит соединение, не переспрашивая DNS повторно на лету (для простого
`proxy_pass http://имя:порт` без специальной настройки `resolver`).

**Решение:** после пересоздания любого сервиса, на который проксирует nginx,
просто перезапустите nginx: `docker restart nginx` (не пересоздание, просто
restart — этого достаточно, чтобы он заново резолвнул имена).

### 6.9. `autossh` не переподключается после "чистого" разрыва

**Симптом:** SSH-туннель (`ssh -L 8080:localhost:8080`) периодически падал.
Поставили `autossh`, ожидая, что он сам переподнимет туннель при любом сбое —
но при тестовой проверке (`kill` дочернего ssh-процесса) `autossh` тоже
завершился, а не перезапустился.

**Причина:** в логе `autossh` было прямым текстом: `ssh exited with status 0;
autossh exiting`. Если дочерний `ssh` завершается "чисто" (код 0 — так бывает
при штатном `kill`, а не только при обрыве сети), `autossh` расценивает это
как намеренную остановку пользователем и не перезапускает процесс. Это
поведение by design, а не баг — но оно не то, что нужно для "туннель должен
жить всегда".

**Решение:** обернуть сам `autossh` в bash-цикл `while true; do autossh ...;
sleep 3; done` — тогда даже если `autossh` решит остановиться сам, внешний
цикл перезапустит и его. Проверили: `kill -9` и на `autossh`, и на дочерний
`ssh` одновременно — туннель поднимается заново за пару секунд.

## 7. Командная работа

- **Ветки:** каждый работает в своей ветке (`feature/...`), мержи в `main` —
  после того как второй человек посмотрел диф (см. README про PR).
- **Кто что трогает:** договоритесь заранее, кто отвечает за сбор/нормализацию
  (`dags/drom_ingest.py`, `dags/telegram_ingest.py`, `dags/normalize_car_listings.py`,
  `common/parsers.py`, `common/llm.py`), а кто — за дашборд/модель/деплой
  (Superset, `prediction_service/`, `docker-compose.yml`, `nginx/`). Общие
  файлы (`docker-compose.yml`, `common/config.py`) трогать через PR с ревью,
  не напрямую на сервере.
- **Общий сервер:** перед `docker compose up -d --build` (пересборка/рестарт
  нескольких сервисов) или ручными изменениями в БД — предупредить другого
  в чате, чтобы не столкнуться посреди чужой сессии в Airflow UI.

## 8. Финальный чек-лист проверки

```bash
# 1. Все контейнеры живы
docker compose ps

# 2. Данные растут
docker compose exec postgres psql -U <user> -d <db> \
  -c "select source, count(*) from car_listings group by source;"

# 3. DAG'и не на паузе и последние раны — success
docker compose exec airflow-scheduler airflow dags list
docker compose exec airflow-scheduler airflow dags list-runs -d drom_ingest

# 4. Дашборд открывается и показывает данные
curl -o /dev/null -w '%{http_code}\n' http://<server>/health

# 5. Сервис предсказания отвечает
curl -X POST http://<server>:81/predict -H "Content-Type: application/json" \
  -d '{"url":"<реальная ссылка на объявление>"}'

# 6. Прямые порты закрыты снаружи (должны таймаутить извне, но открываться
#    через SSH-туннель)
curl --max-time 5 http://<server>:8080/health   # ожидаем: не отвечает
```
